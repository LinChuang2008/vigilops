## 二、传统监控 vs AI 可观测性：差在哪？

| 能力 | 传统监控（Zabbix/Prometheus） | AI 可观测性（VigilOps） |

| 发现问题 | ✅ 阈值告警 | ✅ 异常检测 + 阈值 |

| 历史经验 | ❌ 全靠人记 | ✅ 运维记忆系统 |

**核心差异**：传统监控告诉你"出了问题"，AI 可观测性告诉你"为什么出问题"并"自动修好"。

## 三、自动修复是怎么工作的？

以凌晨 3 点磁盘空间告警为例：

### 传统流程：
```
03:00 告警 → 03:05 值班被叫醒 → 03:15 SSH 登录 → 03:20 df -h 排查
→ 03:30 手动清理日志 → 03:35 确认恢复 → 03:40 回去睡觉（睡不着了）
```
**耗时 40 分钟，人被打断**

### VigilOps 自动修复流程：
```
03:00 告警触发 → AI 分析磁盘使用模式 → 匹配 disk_cleanup Runbook
→ 自动清理过期日志和临时文件 → 03:02 恢复正常
→ 飞书推送："磁盘告警已自动处理，释放 12GB 空间"
→ 经验存入运维记忆，下次更快
```
**耗时 2 分钟，人不用醒**

### 内置 6 个 Runbook：
- `disk_cleanup` — 磁盘清理
- `memory_pressure` — 内存压力缓解
- `service_restart` — 服务重启
- `log_rotation` — 日志轮转
- `zombie_killer` — 僵尸进程清理
- `connection_reset` — 连接重置

每个 Runbook 都有安全检查和审批流，不会"自动修成灾难"。

## 四、运维记忆：越用越聪明

传统监控每次告警都从零开始排查。VigilOps 有**运维记忆系统**：

- 每次故障修复后，自动存储经验
- 下次类似问题，AI 先回忆历史处理方案
- 相当于给你的监控系统装了个"老运维的脑子"

```
第 1 次：磁盘满 → AI 分析 → 发现是 Nginx 日志暴涨 → 修复 + 存储经验
第 2 次：磁盘满 → 直接匹配历史方案 → 30 秒修复
第 N 次：还没满就预警 → 主动清理 → 告警都没触发
```

## 五、15 分钟跑起来

```bash
# 1. 克隆项目
git clone https://github.com/LinChuang2008/vigilops.git
cd vigilops

# 2. 配置环境变量
cp .env.example .env
# 编辑 .env，设置数据库密码和 DeepSeek API Key

# 3. 一键启动
docker compose up -d

# 4. 访问
# 前端：http://localhost:3001
# API：http://localhost:8001/docs
```

4 个容器（后端 + 前端 + PostgreSQL + Redis），资源占用约 1GB 内存。

### 被监控服务器安装 Agent：

```bash
curl -fsSL https://raw.githubusercontent.com/LinChuang2008/vigilops/main/agent/install.sh | bash -s -- \
  --server http://your-vigilops-server:8001 \
  --token YOUR_AGENT_TOKEN
```

60 秒后开始收到指标数据。
